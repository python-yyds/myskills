# 论文类型自适应

当 `paper-reading-zh` 已触发，且论文不是明显的标准方法 / 实验结构，或类型会改变主体章节时读取本文件。

## 目录

- 判断原则与快速判断表
- 共用前置块
- 标准方法 / 实验论文
- 系统 / 测量论文
- 数据集 / benchmark 论文
- 理论 / 证明论文
- 综述 / 立场论文
- 观点 / 路线图变体
- 混合类型与不确定类型

## 判断原则

论文类型回答的是“这篇材料主要靠什么结构建立贡献和证据”，不等同于 venue、研究领域或用户选择的阅读模式。

1. 先看主要贡献，再看主要证据对象。不要只凭标题里的 system、benchmark、theory、survey 等词判断。
2. 优先使用摘要、目录、贡献列表、方法或问题设定、结果章节和结论共同判断。
3. 混合型论文选择一个主类型，并可说明次要类型及判断依据；不要为了分类反复追问用户。
4. 只能看到摘要或局部材料时，按当前可见的主要证据结构选择最接近类型，同时降低结论强度。
5. 仍无法判断时不强行贴类型标签；沿用所选主模式，只使用材料实际支持的章节，不补写不存在的方法、实验或证明。

快速判断：

| 类型 | 主要贡献与证据结构 | 容易混淆的边界 |
|---|---|---|
| 标准方法 / 实验 | 新算法、模型、训练方法或技术机制，主要靠实验比较、消融或分析验证 | 论文使用了新数据集，不代表主贡献是数据集 |
| 系统 / 测量 | 新系统、系统优化、部署经验，或对真实对象进行大规模观测与测量 | 只有一个工程实现但核心贡献仍是算法时，仍选标准方法 / 实验 |
| 数据集 / benchmark | 新数据资源、任务定义、评估协议或基准套件，主要证据是构建质量、覆盖与基线结果 | 只是在已有 benchmark 上刷新分数，不是 benchmark 论文 |
| 理论 / 证明 | 新定理、界、复杂度结果、可识别性或收敛性结论，主要证据是形式化证明 | 带少量定理但主要贡献靠实验时，不选理论 / 证明 |
| 综述 / 立场 | 文献综合、分类框架、研究议程或规范性立场，主要证据是文献覆盖与论证链 | 未来时间表、产品路线或产业预测占主导时，选观点 / 路线图 |
| 观点 / 路线图 | perspective、roadmap、产业 thesis、white paper 等未来导向主张，主要靠案例、趋势、假设和预测 | 有系统化文献选择与综合时，更接近综述 / 立场 |

## 共用前置块

深读模式默认保留：

```markdown
关键词：...

一段话总结：
...

论文基本信息：
- 标题：
- venue/年份：
- 链接：
- 任务领域：
```

下面各类型骨架只替换主体章节。工程拆解模式和调研比较模式仍沿用各自主模式，但要吸收相应类型的重点。按图表顺序子开关触发时，以真实图表顺序组织主体，同时保留类型要求。证据审计触发时，叠加 `modes.md` 的审计字段，不需要复制两套完整输出。

## 标准方法 / 实验论文

### 识别原则

- 核心贡献是算法、模型、训练目标、推理方法或技术机制。
- 主要证据是受控实验、基线比较、消融、敏感性分析或误差分析。
- 论文主线可以自然表述为“问题缺口 -> 方法设计 -> 实验验证 -> 局限”。

### 适用边界

- 新系统只是承载算法实验的平台时，仍按本类型。
- 新数据集只是验证方法的一个实验资源时，仍按本类型。
- 形式化结果只用于辅助解释，而主要贡献依赖实证结果时，仍按本类型。

### 建议主体骨架

```markdown
1. 核心问题与贡献
2. 方法深度解析
3. 实验与结果
4. 批判性讨论
5. 复现/应用提示（条件性省略）
```

实验分析要说明每项结果验证了哪个设计假设，不只复述分数高低。实验部分还需讲清所提方法在实验中的具体实施流程：用了什么数据、什么模型、关键参数如何设置、从输入到输出的完整流水线，可举一个具体运行示例，让读者知道"如果我要复现这个实验，实际需要做什么"。论文未说明的实现细节写"论文未说明"，不靠猜测补齐。

## 系统 / 测量论文

### 识别原则

- 系统论文的核心贡献通常是架构、组件协作、资源管理、可靠性、性能优化或真实部署经验。
- 测量论文的核心贡献通常是对平台、网络、软件生态、用户行为、模型或产业对象进行系统观测，证据来自日志、trace、探针、调查、样本或长期统计。
- 主要可信度取决于工作负载、采样方式、测量工具、基线、时间窗口和有效性威胁，而不只是一组最终指标。

### 适用边界

- 如果系统只是算法实现，论文主张仍由模型实验支撑，选标准方法 / 实验。
- 如果主要产物是可复用数据集或评估套件，选数据集 / benchmark。
- 测量结果不能自动推广到未覆盖平台、人群、地区、版本或时间段。

### 建议主体骨架

```markdown
1. 问题场景与设计目标 / 测量对象
2. 系统架构与关键机制 / 测量方法与采样
3. 工作负载、数据来源、时间窗口与实验设置
4. 关键性能结果 / 主要测量发现
5. 有效性威胁、偏差与可推广边界
6. 部署、复现或工程启示（条件性输出）
```

写法要求：

- 系统论文区分设计目标、机制、实现和评估；不要把实现规模直接写成设计正确性的证明。
- 测量论文说明总体、样本、采集过程、缺失数据和观测偏差；相关性结果不写成因果结论。
- 内部 trace、未公开 benchmark、生产规模或供应链数字按主 skill 的高影响声明标签处理。

## 数据集 / benchmark 论文

### 识别原则

- 核心贡献是数据集、任务定义、标注体系、评估协议、指标或 benchmark 套件。
- 主要证据是数据构建过程、质量控制、覆盖统计、一致性、基线结果和评估有效性。
- 模型通常用于验证资源或协议，而不是论文的唯一主贡献。

### 适用边界

- 仅使用新数据验证一个新模型时，选标准方法 / 实验。
- 只汇总已有公开数据且没有新的任务、协议或质量控制贡献时，未必构成数据集 / benchmark 类型。
- 高基线分数或低基线分数都不能单独证明 benchmark 有效。

### 建议主体骨架

```markdown
1. 资源要填补的任务缺口与预期用途
2. 数据来源、采集、清洗、标注与治理
3. 任务定义、数据划分、指标和评估协议
4. 基线系统与结果能验证什么
5. 覆盖、泄漏、污染、偏差与有效性风险
6. 许可、维护、适用边界与潜在误用
```

写法要求：

- 核对数据许可、隐私或同意、标注者信息、质量控制和版本维护；论文未说明时明确写出。
- 检查训练集与测试集泄漏、近重复、预训练污染、指标饱和、测试集过拟合和基线预算是否可比。
- 不把 benchmark 排名直接等同于真实场景能力，也不把数据规模直接等同于数据质量。

## 理论 / 证明论文

### 识别原则

- 核心贡献是定理、上下界、复杂度、收敛性、可识别性、最优性、不可行性或形式化性质。
- 主要证据是定义、假设、引理、定理和证明之间的依赖链。
- 实验若存在，通常用于说明现象或展示适用性，不替代证明。

### 适用边界

- 只有常规收敛说明或附录定理，但主要贡献依赖实证结果时，选标准方法 / 实验。
- 形式化模型与真实系统之间的映射需要单独讨论；定理成立不自动证明工程场景有效。
- 材料缺页、公式乱码或证明不可读时，不重构缺失推导。

### 建议主体骨架

```markdown
1. 问题设定、定义与符号
2. 核心假设和主要定理
3. 定理直觉与证明策略
4. 关键引理、证明依赖与难点
5. 界的紧致性、反例与适用范围
6. 对算法、实验或实际问题的含义
```

写法要求：

- 区分已证明结论、证明所需假设、作者直觉和自己的解释。
- 解释假设何时可能不成立，以及结论对参数、分布、规模或计算模型的依赖。
- 只能确认证明轮廓时就写“当前材料只能确认证明轮廓”，不要补全论文没有展示的步骤。

## 综述 / 立场论文

### 识别原则

- 综述的核心贡献是定义范围、选择文献、建立分类框架、综合共识与分歧、总结开放问题。
- 立场论文的核心贡献是提出规范性判断、研究议程或应采取的方向，主要靠论点、引用、案例和反方处理建立可信度。
- 证据重点是材料覆盖、选择方法、论证链和替代观点，不是强行寻找模型消融。

### 适用边界

- 未来时间表、产品路线、产能或供应链预测占主导时，选观点 / 路线图变体。
- 文献列表很长不等于系统综述；没有检索与筛选方法时，应说明覆盖边界未知。
- 引用一篇外部论文只说明作者采用了该来源，不等于当前材料已经独立复核其结论。

### 建议主体骨架

```markdown
1. 覆盖范围与核心问题 / 核心立场
2. 文献选择方法与材料边界 / 论证链
3. 分类框架、共识、分歧与关键论据
4. 证据覆盖与证据强度
5. 遗漏文献、反方观点与作者偏向
6. 研究议程、实践含义与开放问题
```

写法要求：

- 对综述说明时间范围、数据库、关键词、纳排标准是否公开；未公开时不要称为穷尽性综述。
- 对立场论文区分事实判断、价值判断、政策或工程建议，并检查是否认真处理反例和替代解释。
- 不按被引次数或作者声望替代对论证质量的分析。

## 观点 / 路线图变体

### 识别原则

- 适用于 perspective、roadmap、产业 thesis、未来导向 white paper，以及以趋势判断、阶段目标或预测为主的材料。
- 核心不是可复现实验或系统化文献综合，而是主张链、工程与产业假设、案例、趋势外推和未来路径。

### 适用边界

- 有明确检索、筛选与文献综合方法时，优先按综述 / 立场。
- 有可运行系统和受控性能评估时，优先按系统 / 测量。
- 路线图中的日期、产品状态和产能数字不能因为写在论文中就视为已实现事实。

### 建议主体骨架

```markdown
1. 核心主张链条
2. 证据类型与证据强度
3. 关键产业 / 工程假设
4. 可核验、未核验与作者预测
5. 风险边界与适合追问的问题
```

写法要求：

- 少写“方法深度解析”，重点解释主张如何由背景、案例、数据和预测支撑。
- 产品路线图、未发布芯片、内部 benchmark、供应链能力、生产数量等高影响声明，必须标注“论文内部声明 / 作者预测 / 已公开第三方核验 / 未核验”。
- 没有公开复验材料时，不把作者声明写成已独立证明的事实。

## 混合类型与不确定类型

- 系统 + 测量：如果贡献是系统设计，按系统骨架组织，把测量方法放入评估；如果贡献是生态发现，按测量逻辑组织，把工具视为采集方法。
- 数据集 + benchmark：通常可合并使用同一骨架，但要分别评价数据质量和评估协议。
- 方法 + 理论：按最终主张依赖的证据选择。主要靠定理成立选理论，主要靠实验证明有效选标准方法 / 实验。
- 综述 + 观点：有明确文献选择与综合方法选综述 / 立场；未来路径、产业假设和预测占主导选观点 / 路线图。
- 能判断最接近类型时，说明“按当前材料暂归为 X 类型”并完成相应骨架；仍无法判断时不贴类型标签，沿用主模式并只保留材料支持的章节，不重复要求用户分类。
